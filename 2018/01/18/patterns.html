常用架构模式

2018-01-18

架构模式

用于描述软件系统里的基本的结构组织或纲要。架构模式提供了一些定义好的子系统，指定它们的责任，并给出把它们组织在一起的法则和指南。

好架构模式的特点？

  • 任务均衡分摊给具有清晰角色的实体。
  • 高内聚、低耦合。
  • 高重用性、低维护成本。
  • 可单独测试。

一、MVC模式

1.1.功能划分

模型层M

封装数据、处理数据的业务逻辑。

视图层V

显示与用户交互的界面、响应用户的操作。

控制器C

①响应视图层上的用户操作。

②业务逻辑的调用，触发M层的更新。

③M层数据发生变化时通知控制器来更新它负责的视图。

 
传统的 MVC 模式中，三层分工明确，但是三层之间相互都有通信，耦合度高、复用性低。Apple 在 iOS 开发中对传统 MVC 进行了改进，根据苹果文档的描述，理想中这种模式下，视图层与模型层不应有直接的关联。视图层将用户的操作通知给控制器，控制器负责更新模型层。而当模型层发生变化时，会通过KVO等机制通知到控制器并由它更新对应的视图。

1.2.实际开发中的MVC

技术上来讲视图层与控制器是相互独立的，但实际开发过程中它们总是结对出现、高度耦合。比如：控制器要负责维护视图的生命周期，响应用户在视图层的操作，实现诸如Tableview等视图的代理等。。所以，上面的情况就造成一种现象：典型的 MVC 事实上变成了 M-VC。

再者，理想的MVC模式中cell不引用模型对象，那么cell的设置就要放到控制器中，这样就会增加控制器的代码量。其他诸如网络请求、工具方法等显然不适合放在视图和模型层里，而一旦放在控制器里，同样也会造成控制器的臃肿。这些就是Massive View Controller的由来。。

当业务逻辑和展示逻辑都在Controller中时，就很难进行单元测试。

二、MVP模式

2.1.功能划分

模型层M

功能与 MVC 中的Model类似。

视图层V

由view和Controller组成，主要作用：

①界面元素布局和动画。

②将用户操作事件交给P层处理并展示处理结果。

③P层中的业务逻辑处理完毕后通过代理告知V层刷新界面。

④Controller负责生成视图，实现视图的代理和数据源以及界面的跳转等。

控制层P

扮演着视图层与模型层中间人的角色：

①暴露接口给视图层的事件调用并实现视图层的事件处理逻辑。

②调用模型层的接口获取数据，加工数据并将其封装成视图层适用的数据和状态，实现view与Model的解耦。

 
MVP 是由 MVC 演化而来的，它对 MVC 中的Controller层进行了优化而生成了Presenter。P层和 MVC 中的Controller一样，负责核心逻辑。但不一样的是P层通过接口和协议进行数据传递，从而使 View 和 Model 解耦，更加专注于自身业务逻辑。同时，因为主要的逻辑在P层，所以可以方便的对其进行单元测试。

使用 MVP 会增加代码量，但是总体上减少了代码的复杂性。

三、MVVM模式

3.1.功能划分

模型层M

与 MVP 中的 Model 类似。

视图层V

由 View 和 Controller 组成，负责显示 UI。绑定 ViewModel 中的属性，触发 ViewModel 中的命令。

视图模型层VM

负责用户输入验证逻辑，视图显示逻辑，发起网络请求、从 Model 中获取V层所需的数据，转换成V层可以展示的数据，并暴露公开的属性和命令供V层进行绑定。

Binder

在 MVVM 中，声明式的数据和命令绑定是一个隐含的约定，它可以让开发者非常方便地实现 V层和ViewModel的同步，避免编写大量繁杂的样板化代码。（ReactiveCocoa）

 
MVVM 是微软提出来的，它其实就是在 MVP 的基础上发展起来的。因此，MVVM 各层的职责基本上与 MVP 的类似，其中VM对应P层。那么 MVVM 相对于 MVP 改良了什么呢？答案就是 数据绑定。

在 MVP 中，从用户点击开始，一个完整的响应流程是：V层调用P层处理业务逻辑，P层调用M层处理数据，处理完成后 P层再通过代理回调V层。这里要反复同步P层的状态，当事件多起来时这样写就有点麻烦。
 
而在 MVVM 中， View与VM层之间多了数据绑定的操作，这意味着当你更新VM层的数据的时候，V层会相应的变动 UI，自动实现状态的同步。可参考此文中的代码来帮助理解。

MVVM 模式中，ViewModel存在的目的在于抽离Controller中的展示业务逻辑，而不是代替Controller本身。所以它不负责视图的操作，也就不需要持有任何的视图对象，更不能包含视图的push/present等跳转逻辑。这里，Controller 要做的仅是将视图和ViewModel进行绑定，同时管理各个视图。所以实际上我们的 MVVM 也可以看成M-VM-C-V。
