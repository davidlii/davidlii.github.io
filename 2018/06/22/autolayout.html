布局与约束相关知识点

2018-06-22

整理自#©zpz5789-Updating constraints、Layout

#1、布局相关的接口

系统重写布局，若需要更精确控制子view，就需要重写该方法。此方法会在以下场景下触发：

  • 使用-init初始化时不会触发。
  • 用-initWithFrame 初始化且 rect 的值不为 CGRectZero 时会触发；
  • 调用 addSubview 时会触发；
  • 设置 view 的 frame 且 frame 的值设置前后发生了变化时会触发；
  • 旋转屏幕会触发父 UIView 上的 layoutSubviews 事件；
  • 改变一个视图大小时也会触发父视图上的 layoutSubviews 事件。

此方法由系统自动调用，不建议我们直接调用此方法。如果有需要时，我们可以通过 [view layoutIfNeeded] 的方式来让系统自动触发-layoutSubviews方法。

强制立即更新页面布局，从当前 view 开始，此方法会遍历整个 view 层次（包括superviews）请求重新布局。因此调用此方法会强制整个 view 层次布局。如果希望立刻生成新的 frame 需要调用此方法，利用这点一般布局动画可以在更新布局后直接使用这个方法让动画生效。

告知页面需要更新，但是不会立刻开始更新。此方法会将 view 当前的 layout 设置为无效的，并在下一个 upadte cycle 里去触发 layout 更新。执行后会立刻调用 layoutSubviews。

若需要在当前 view 上绘制自定义的内容，则需要重写此方法。

将视图标记为需要重绘，实质上是 layer 的重绘，最终触发-drawRect:方法。

#2、约束相关的属性&接口

除了AutoLayout，Autoresizing 也是一种布局方式。默认情况下 translatesAutoresizingMaskIntoConstraints ＝ YES , 此时视图的 AutoresizingMask 会被转换成对应效果的约束。这样很可能就会和我们手动添加的其它约束有冲突。此属性设置成 NO 时，AutoresizingMask 就不会变成约束，也就是说当前视图的 AutoresizingMask 失效了。
 

手写代码添加视图时，此属性的值默认为 YES，因为我们需要手动添加和修改约束，所以需要将此属性置为 NO。使用 IB 创建视图时，系统会自动将此属性置为 NO。

“Constraint-based layout system” 使用此返回值去决定是否需要调用 updateConstraints 作为正常布局过程的一部分。

系统更新约束，自定义 view 时可以重写此方法，在其中添加 view 需要的局部 contraints。注意：要在实现在最后调用 [super updateConstraints]。

立即触发约束更新，自动更新布局。

当有约束需要更新时，调用此方法来标记这些约束需要在未来的某个点更新，系统之后调用 updateConstraints。

#3、(Auto Layout)自动布局的过程

updating constraints -> layout -> display，每一个步骤都依赖于上一步。
 

(1) updating constraints：从下向上（from subview to superview），为下一步layout准备信息。可以通过调用方法 setNeedUpdateConstraints 去触发此步。constraints 的改变也会自动的触发此步。但是当你自定义 view 时，如果一些改变可能会影响到布局时，需要自己去调用 updateConstraintsIfNeeded 以通知自动布局更新约束。
 

(2) layout：从上向下（from superview to subview），此步主要应用上一步的信息去设置 view 的 center 和 bounds。可以通过调用 setNeedsLayout 去触发此步骤，此方法不会立即应用 layout。如果想要系统立即的更新 layout，可以调用 layoutIfNeeded。另外自定义 view 时可以重写 layoutSubViews 方法来得到更多的定制化效果。
 

(3) display：从上向下（from superview to subview）。此步是把 view 渲染到屏幕上，它与你是否使用 Auto layout 无关，通过调用 setNeedsDisplay 触发，系统会调用 UIView 的 drawRect 方法。

#4、VC中视图的布局过程

  • -loadView
  • viewDidLoad
  • viewWillAppear
  • updateViewConstraints
  • viewWillLayoutSubviews
  • viewDidLayoutSubviews
  • viewDidAppear
  • viewWillDisAppear
  • viewDidDisAppear

注意，这个-updateViewConstraints方法是 UIViewController 中的，不是 UIView 中的-updateConstraints方法。。“view 属于某个 VC”（比如 self.view），当此 view 需要执行updateConstraints时，会调用 VC 的updateViewConstraints，这样就不用你再自定义此 view 并重写其updateConstraints方法了。方法中需要调用其 super 方法。这个方法的默认实现是调用其所有 subview 的 updateConstraints 方法，这样就自上而下的完成了布局。
 

另外，updateViewConstraints、viewWillLayoutSubviews、viewDidLayoutSubviews 在当前 vc 布局过程中可能会被多次调用，这三个方法在离开当前控制器时也会被调用。

#5、小结：

除了self.view外，其他视图的实际 frame 要在以下两个方法执行完之后才能确定：

  • view 的 layoutSubViews;
  • VC 的 viewDidLayoutSubviews;

所以，当我们真的需要 frame 时要在这个两个时间点以后再去获取。这也可以解释为什么 viewDidLoad 里通过setFrame的方式修改原先在xib里拖动的约束代码无效了。因为 updateViewConstraints 在 viewDidLoad 后执行，会覆盖掉之前的设置的frame，所以无效。
 

另外需要注意一点：在使用 sb 或者 xib 加载 VC 时，viewDidLoad 中只有 self.view 的 frame 信息是准确的，其他所有的控件的 frame 都是它们在 sb 或 xib 中的原始尺寸，与当前设备可能不符。
