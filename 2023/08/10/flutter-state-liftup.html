  2. 通过setState告诉框架：我已修改，你去重新绘制整个组件树；

2.提升状态

如前文所述，原生开发中修改某组件的状态，通常是拿到该组件的对象并直接修改其属性值。

而响应式框架中，获取别的组件不是很方便，组件之间一般只能向上或者向下通信，兄弟组件之间无法直接通信。

为此，Flutter采用了「Lift state up」理念：将数据从子组件移到某个父组件，以便在那里使用它，再将其传递给其他子组件。例如，为实现AB组件间的通信：

  1. 需要沿着组件树传递A的state；
  2. 或者提升state到必要的高度，直到B组件能获得这份数据；
  3. 或者AB组件拥有共同的父组件。

3.性能问题

如果使用不当，提升状态可能造成一些不必要的性能开销：

拿前文的计算器案例来说，点击加号按钮后，按道理只需重绘Text一个节点即可，其他组件无需重绘。但_counter状态在父节点MyHomePage中，setState时整个MyHomePage全都重新绘制了，即appBar、底部按钮这些组件都重绘了。这种大范围重绘引起的部分组件不必要的重绘，显然是一种浪费性能的行为。

这说明，DakBox在MyHomePage重绘时跟着重新 build 了一遍，而DakBox并不涉及_counter状态，没必要重新 build 以节省性能。

结合常见案例来看，这种性能问题多是由【大范围重绘】与【组件不必要的重绘】造成的，那么解决问题的思路就可以从这两方面入手：

  1. 缩小重绘范围；
  2. 禁止组件参与重绘。

下面还是以计算器案例来演示解决这种性能问题的三种方案~

解决方案1

将实例DakBox的创建提到State所在组件的initState阶段，保证只创建一次。

但这种做法不值得推荐~因为在Flutter框架的共识中，所有跟UI相关的代码，都应该写在 build 中，这样在setState时才能保证绘制出来的东西是正确的。另外如果创建DakBox()实例时需要传入context这样的参数，在 initState 阶段就很不方便传入了。

解决方案2

使用const修饰DakBox实例，确保只绘制一次。

DakBox只在MyHomePage第一次 build 时重绘了一次，之后点击按钮更新数值时不再重新绘制。

此方案的不便：无法给const组件传入Theme.of()或MediaQuery.of()等需要动态获取的值~

解决方案3

借助特殊机制，只在小范围内重绘组件。

这里，我们使用了ValueNotifier，它是一个类似于Swift中属性包装器@propertyWrapper的东西，内部为我们封装了属性操作和发送通知等业务逻辑，这个在后面的文章中再详细介绍。
